# RFC: Custom Game Events

**Issue:** labelle-toolkit/labelle-engine#422  
**Status:** Draft  
**Author:** Alexandre  
**Date:** 2026-04-07

## Problem

Scripts that need to react to game-specific state changes must poll ECS components every frame. This couples visual effect scripts to logic scripts and wastes cycles checking state that rarely changes.

**Example:** When a worker goes to sleep, `worker_animation.zig` must poll `NeedState` every frame, detect the transition to `fulfilling + Sleep`, manage curtain entity lifecycle, track floor positions, and handle the wake-up transition — all in a script whose job is sprite switching.

The engine has a hook system (`HookDispatcher` + `HookPayload` union), but it only supports built-in lifecycle events (`game_init`, `scene_load`, `entity_created`, etc.). Games cannot define custom events.

## Proposal

Allow games to define custom event types in an `events/` directory. The CLI discovers them and merges them into the hook payload union. Scripts can emit and listen to these events using the existing hook dispatch infrastructure.

### Game-side API

#### 1. Define events (`events/worker_events.zig`)

```zig
pub const WorkerSleepStart = struct {
    worker_id: u64,
    bed_id: u64,
};

pub const WorkerSleepEnd = struct {
    worker_id: u64,
    bed_id: u64,
};
```

#### 2. Emit events (in any script)

```zig
// In needs_manager, when worker arrives at bed:
game.emit(.{ .worker_sleep_start = .{
    .worker_id = worker_id,
    .bed_id = bed_id,
} });

// When fulfillment completes:
game.emit(.{ .worker_sleep_end = .{
    .worker_id = worker_id,
    .bed_id = bed_id,
} });
```

#### 3. Listen to events (in any script)

Scripts declare an `onEvent` function following the existing convention pattern:

```zig
// In curtain_animation.zig:
pub fn onEvent(game: anytype, state: anytype, event: anytype) void {
    switch (event) {
        .worker_sleep_start => |e| startCurtainClose(game, state, e.bed_id),
        .worker_sleep_end => |e| startCurtainOpen(game, state, e.bed_id),
        else => {},
    }
}
```

### Event delivery

Events are **buffered within the current frame** and delivered after the emitting script's tick completes (or at end of frame). This avoids re-entrant script execution and ordering issues.

Two delivery options to consider:

**Option A: End-of-frame delivery.** All events emitted during the frame are delivered to all `onEvent` handlers after all script ticks complete. Simple, predictable, but introduces one-frame latency.

**Option B: Inter-script delivery.** Events emitted by script N are delivered to scripts N+1..last before their tick runs. Events emitted in the last script are delivered next frame. No latency for scripts ordered after the emitter.

## Changes Required

### Engine (`labelle-engine`)

1. **Event buffer** — Add a frame-scoped event queue to the game struct. `game.emit(payload)` appends to the buffer.

2. **Event dispatch** — After script ticks (or between scripts), iterate the event buffer and call `onEvent` on scripts that declare it. Clear the buffer after delivery.

3. **Merged payload type** — Extend `MergeHookPayloads` (or create `MergeEventPayloads`) to combine engine hook events with game-defined event types. The game's event payload union is generated by the CLI.

4. **Script runner** — Detect `onEvent` in scripts via comptime reflection (same pattern as detecting `tick`, `init`, `deinit`). Dispatch events to matching scripts.

### CLI (`labelle-cli`)

1. **Scanner** — Add `events/` directory scanning (same pattern as `components/`, `hooks/`). Each `.zig` file is scanned for public struct types.

2. **Codegen** — Generate a `GameEvents` tagged union from discovered event structs:
   ```zig
   const GameEvents = union(enum) {
       worker_sleep_start: WorkerSleepStart,
       worker_sleep_end: WorkerSleepEnd,
   };
   ```
   Field names derived from struct names (PascalCase to snake_case).

3. **Template** — Add `{{game_events_block}}` to `main.zig.template` for the generated union and event queue wiring.

### Core (`labelle-core`)

No changes. `HookDispatcher` and `MergeHookPayloads` are reusable as-is for the dispatch mechanism.

## Alternatives Considered

### Marker components

Add/remove a `Sleeping` component as a signal. The listening script queries for entities with the marker.

- Pro: Works today, no engine changes
- Con: Pollutes ECS with transient markers, still requires per-frame polling to detect add/remove

### Shared state via script State structs

Scripts can't access each other's state — by design. Not viable.

### Extend HookPayload directly

Add game events as new variants in the engine's `HookPayload` union.

- Con: Game-specific events don't belong in the engine
- Con: Every game would bloat the shared union

## Migration

Existing games are unaffected — no `events/` directory means no custom events, no codegen changes. The feature is purely additive.

## Open Questions

1. **Delivery timing:** Option A (end-of-frame) vs Option B (inter-script)? Option A is simpler but adds one-frame latency.
2. **Event ordering:** Should events be delivered in emit order, or grouped by type?
3. **Filtering:** Should scripts declare which events they care about (like ECS views), or receive all events and switch?
4. **Persistence:** Should events survive scene transitions? Probably not — clear on scene unload.
